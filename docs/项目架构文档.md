# 项目架构文档

## 1. 项目目标

本项目是一个基于 Vue 3 + Vite + Node Relay 的应用，用于演示并实现以下能力：

1. 支持上传本地 wav，或使用预留 wav 文件。
2. 在网页中调用 Qwen 声音复刻接口，生成专属 `voice`。
3. 调用 Qwen3 Realtime TTS（WebSocket）完成文本到语音合成。
4. 将返回的音频流拼接后在网页中直接播放。

项目当前采用“前端 + relay 转发服务”模式，避免浏览器直连 WebSocket 的兼容问题。

---

## 2. 总体架构

```text
用户浏览器
  ├─ Vue 页面（参数配置、文件上传、日志展示、音频播放）
  └─ HTTP 调用
     └─ 本地 Relay 服务
        ├─ /clone -> 声音复刻 API
        └─ /synthesize -> Realtime WebSocket -> 音频回传
```

### 2.1 分层说明

1. 表现层（UI）
   - 表单输入：API Key、地域、模型、合成模式、文本内容。
   - 文件输入：上传 wav 或读取预留 wav。
   - 结果展示：日志区 + 音频播放器。

2. 业务层（前端逻辑）
   - 音色复刻：`createVoiceFromWav()`。
   - 语音合成：`synthesizeSpeech()`。
   - 流式音频组装：`base64ToBytes()` + `Blob`。

3. relay 服务层（Node + ws）
   - 统一处理上游鉴权。
   - 将 WebSocket 合成能力封装为 HTTP 接口。

4. 外部服务层（DashScope）
   - REST：声音复刻。
   - WebSocket：实时语音合成。

## 4. 核心实现流程

## 4.1 流程 A：生成 voice（声音复刻）

1. 获取 wav 文件
   - 上传模式：使用文件输入框中的 wav。
   - 预留模式：通过 `fetch('/reserved.wav')` 获取静态文件。

2. 将 wav 转换为 Data URI
   - 使用 `FileReader.readAsDataURL()`。

3. 调用声音复刻 REST API
   - 接口：`/api/v1/services/audio/tts/customization`
   - 请求体关键字段：
     - `model: qwen-voice-enrollment`
     - `target_model: 当前选择的 TTS 模型`
     - `audio.data: data URI`

4. 解析返回
   - 从 `output.voice` 提取 voice id 并保存到页面状态。

## 4.2 流程 B：文本转语音（通过 relay）

1. 前端调用 relay `/synthesize`
   - 传入 `region`、`model`、`voice`、`text`、`mode`、`apiKey`。

2. relay 连接上游 Realtime WebSocket
   - 根据 `region` 选择 ws endpoint。
   - 在 Header 中设置 `Authorization: Bearer <apiKey>`。

3. relay 向上游发送事件
   - `session.update`：设置 `mode`、`voice`、`response_format`、`sample_rate`。
   - `input_text_buffer.append`：发送待合成文本。
   - 若是 `commit` 模式，再发送 `input_text_buffer.commit`。
   - 最后发送 `session.finish`。

4. relay 处理上游事件
   - `response.audio.delta`：持续接收 base64 音频分片。
   - `response.done`：当前响应完成。
   - `session.finished`：会话结束。

5. relay 回传音频给前端
   - 合并为 mp3 二进制。
   - 前端使用 `Blob` + `URL.createObjectURL` 播放。

## 5. 关键设计决策

1. 为什么先复刻再合成
   - 满足“根据提供 wav 合成”的需求。
   - 允许生成专属 voice 后复用。

2. 为什么同时支持上传和预留 wav
   - 上传模式适合临时体验。
   - 预留模式适合固定演示和自动化测试。

3. 为什么使用 WebSocket
   - Realtime TTS 是流式输出，WebSocket 更适合低延迟收音频分片。

4. 为什么使用 relay 转发
   - 浏览器原生 WebSocket 难以稳定携带鉴权 Header。
   - 通过 relay 可统一处理鉴权与错误。
   - 前端逻辑更简单，播放链路更稳定。

## 8. 风险与限制

1. 当前 UI 仍由用户输入 API Key
   - 生产建议改为服务端环境变量托管。

2. 模型一致性约束
   - 声音复刻的 `target_model` 与 TTS `model` 必须一致。

3. 网络与跨域环境差异
   - 部分网络环境可能影响 WebSocket 连接稳定性。

---

## 9. 可扩展方向

1. 增加“最小后端代理”
   - 用于隐藏 API Key，提升生产可用性。

2. 增加 wav 预处理
   - 自动校验采样率、时长、声道，提升复刻成功率。

3. 增加会话可观测性
   - 记录事件耗时（首包延迟、总耗时、音频长度）。

4. 增加历史记录
   - 保存 voice id、文本和音频结果，便于回放。
